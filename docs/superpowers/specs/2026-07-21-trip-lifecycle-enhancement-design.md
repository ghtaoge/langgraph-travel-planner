# LangGraph Travel Planner 旅行全生命周期增强设计

日期：2026-07-21

## 1. 背景

项目已经具备多阶段 LangGraph 编排、方案选择、逐日确认、整单审批、SSE 流式输出、PostgreSQL 持久化、Checkpoint 回退、用户认证、地图和图片导出等能力。

当前主要问题不是缺少流程节点，而是产品仍停留在“一次性生成行程”的演示阶段：天气、餐饮、酒店和景点工具使用本地模拟数据；行程缺少稳定的业务实体与版本；生成后只能整单反馈，无法可靠地编辑、执行或局部重规划；地图和预算也没有形成可验证的数据闭环。

本设计将项目升级为面向国内自由行的旅行全生命周期助手。建设主线是用户闭环，底层采用可替换 Provider，并只在适合非确定性推理的环节使用 Agent。

## 2. 目标与范围

### 2.1 目标用户

计划国内单城市或城市周边 2 至 7 天自由行的个人用户。

### 2.2 产品目标

- 支持从需求澄清、方案比较、行程编辑到旅中执行和旅行复盘的完整流程。
- 使用真实天气、POI、地理编码和路线数据提高行程可信度。
- 允许用户按活动、当天或未执行行程进行局部重规划。
- 确保锁定项、固定预订和已完成项目不会被 AI 擅自改动。
- 清晰显示数据来源、更新时间、估算状态和降级状态。
- 通过 LangGraph 展示长期状态、人工确认、事件驱动和可恢复执行能力。

### 2.3 首期非目标

- 不提供机票、酒店、门票的真实下单或支付。
- 不支持多人协作、投票和权限管理。
- 不支持复杂多城市调度。
- 不覆盖签证、汇率、时区和跨境合规。
- 酒店、机票和门票价格可以使用 Mock 或估算数据，但必须明确标识。

## 3. 核心用户流程

### 3.1 需求采集

用户通过对话和结构化控件提交目的地、日期、人数、预算、交通偏好、旅行节奏、兴趣及特殊要求。系统将需求解析为结构化约束；目的地、日期、人数等关键字段缺失或冲突时，流程进入澄清节点，不使用模型猜测。

### 3.2 候选方案比较

系统生成三套差异明确的方案，例如经典紧凑、悠闲深度和预算友好。每套方案统一展示预计费用、通勤时间、步行量、每日强度、室内外比例、特色体验和主要风险。

### 3.3 可编辑行程

用户选定方案后创建正式 `Trip`。用户可以拖动活动、删除或新增活动、替换景点，并锁定已预订或不希望 AI 修改的项目。日期、活动、住宿点和交通段均使用稳定 ID。

### 3.4 局部重规划

用户可选择调整单个活动、某一天或所有未执行行程。系统只生成受影响范围内的结构化补丁，保留锁定项、固定预订和已完成项。修改先以差异形式展示，用户确认后才写入。

### 3.5 旅中执行

旅中模式按当天时间轴显示下一项活动、建议出发时间、路线、天气风险和预算。用户可以完成、跳过或延后活动。天气变化、路线耗时异常和景点不可用只生成调整建议，不自动修改行程。

### 3.6 预算与复盘

系统分别记录预计费用和实际支出，按住宿、餐饮、交通、门票等分类。旅行结束后生成计划偏差、预算偏差与偏好总结，并将可复用偏好写入用户长期记忆。

## 4. 总体架构

系统继续使用 Vue 3、FastAPI、LangGraph 和 PostgreSQL，并明确划分为领域层、Provider 层、编排层和前端工作区。

### 4.1 领域层

`Trip` 是业务聚合根。PostgreSQL 中的 Trip 是已确认业务事实，LangGraph State 是一次工作流的执行上下文，聊天消息只作为交互记录。Checkpoint 不作为行程数据的唯一来源。

核心实体包括：

- `Trip`：旅行基础信息、状态、约束和当前版本。
- `TripDay`：日期、每日摘要、活动和住宿。
- `Activity`：稳定 ID、时间、位置、费用、来源、可用状态和执行状态。
- `TransportLeg`：相邻活动间的交通方式、距离、耗时和路线数据。
- `BudgetItem`：分类、预计金额、实际金额和关联活动。
- `TripEvent`：用户操作或实时数据变化产生的事件。
- `PlanRevision`：版本号、结构化补丁、修改原因和确认信息。

### 4.2 Provider 层

业务节点仅依赖统一接口，不直接调用第三方 SDK：

- `WeatherProvider`
- `GeocodingProvider`
- `PoiProvider`
- `RouteProvider`
- `LodgingProvider`
- `PricingProvider`

首期地图、地理编码、POI、路线和天气默认接入高德，以减少密钥、配额和错误模型的种类；和风天气只作为后续可替换实现，不属于首期交付。住宿与票价使用 Mock Provider。Provider 的选择通过配置完成，不影响领域层和图节点。

统一返回结构包含：

```text
data + provider + fetched_at + confidence + stale + estimated + warnings
```

### 4.3 LangGraph 编排层

将现有单条主流程拆为两个职责清晰的图。

`PlanningGraph` 负责首次规划：

```text
collect_requirements
  -> clarify_if_needed
  -> research_destination
  -> generate_candidates
  -> enrich_with_routes
  -> estimate_budget
  -> validate_candidates
  -> user_select
  -> persist_trip
```

`TripOperationGraph` 负责行程变更：

```text
receive_event
  -> classify_scope
  -> load_trip
  -> protect_constraints
  -> generate_patch
  -> validate_patch
  -> compare_revision
  -> user_confirm
  -> apply_revision
```

研究、候选生成和调整建议可以使用专业 Agent。路线计算、预算汇总、时间冲突、锁定项检查、版本控制和持久化必须由确定性代码完成。

### 4.4 前端工作区

现有聊天入口继续保留，并新增围绕 Trip 的旅行工作区：

- 桌面端左侧为日期导航、强度、费用和异常提示。
- 中间为可编辑的每日时间轴。
- 右侧为地图、路线、活动详情和预算面板。
- AI 对话作为辅助入口，不承载全部结构化操作。
- 移动端使用“今天、行程、地图、预算”四个标签页。

现有 `ChatView`、`PlanSelector`、`ItineraryCard` 和 `MapView` 逐步迁移，不要求一次性重写。

## 5. 变更模型与一致性

Agent 输出 `TripPatch`，不得直接覆盖 Trip。补丁至少包含：

- 基准版本号。
- 调整范围。
- 新增、删除、移动和替换操作。
- 交通与预算变化。
- 修改原因与受影响项目。
- 使用的数据来源和更新时间。
- 无法满足的约束与警告。

应用补丁前执行确定性校验：

- 锁定项、固定预订和已完成项未被修改。
- 活动时间不冲突，并保留必要交通时间。
- 每日开始、结束时间和步行上限符合约束。
- 预算与用户容忍范围一致。
- 路线和 POI 数据具备来源，估算数据已标识。

补丁通过校验后，前端展示新增、移除、时间变化、路线变化、预算变化和未满足约束。用户确认后使用乐观锁创建新版本。若基准版本已变化，则拒绝写入、重新加载最新 Trip 并重新生成补丁。历史版本保留且可回退。

## 6. 数据缓存与降级

- 地理编码和 POI 基础信息采用较长缓存。
- 路线按起点、终点、交通方式和时间窗口缓存。
- 天气采用短缓存，并在接近出行日期时提高刷新频率。
- Mock 住宿和价格数据始终标记为 `estimated`。

错误处理分为四级：

1. 单个 Provider 超时：有限重试后读取缓存。
2. 缓存过期：允许返回，但标记 `stale` 并提示更新时间。
3. 服务不可用：切换 Mock 或启发式估算，同时降低置信度。
4. 核心数据不足：停止自动重规划，明确提示缺失数据，不让模型补造事实。

图节点记录节点名、Provider、错误类别、重试性和用户可见说明。任何失败都不能覆盖已确认 Trip；未成功提交的补丁直接废弃。第三方 API Key 只保存在后端，日志过滤密钥、JWT 和精确个人信息。

## 7. 关键交互

方案选择使用统一指标横向比较，避免只比较自然语言描述。

编辑行程时，时间轴与地图双向联动。地图绘制 Provider 返回的真实路线，不使用景点间直线代替路线。活动支持拖动、锁定、删除、替换、完成、跳过和延后。

AI 修改统一进入差异预览：

- 新增项使用绿色标识。
- 移除项使用红色标识。
- 时间、路线和费用变化使用黄色标识。
- 单独列出保留的锁定项。
- 显示修改理由、数据来源和更新时间。
- 提供“应用修改”和“继续调整”操作。

旅中模式只突出当天必要信息和快捷操作，避免把桌面工作区完整压缩到手机屏幕。

## 8. 测试与可观测性

### 8.1 测试策略

- 领域单元测试覆盖 TripPatch、锁定项保护、版本冲突、预算汇总、时间冲突和范围计算。
- Provider 契约测试确保真实与 Mock 实现返回一致结构，并覆盖超时、缓存、过期和降级。
- 第三方服务测试使用录制响应，CI 不依赖实时网络。
- LangGraph 路径测试使用固定模型输出，覆盖澄清、首次规划、用户拒绝、局部重规划、Provider 故障和断点恢复。
- API 测试覆盖 Trip CRUD、版本提交、冲突响应和 SSE 恢复。
- Playwright 覆盖桌面与移动端的规划、编辑、差异确认和旅中执行闭环。

### 8.2 质量评估集

建立不少于 20 个国内自由行案例，覆盖亲子、老人、低预算、雨天、行动不便、高强度和临时延误。自动校验硬约束，并持续记录路线合理性、预算偏差、修改保留率和失败率。

### 8.3 可观测性

每次运行记录 Provider 延迟、缓存命中率、模型调用次数、Token、节点耗时、失败原因和补丁确认率。使用 `trip_id`、`thread_id` 和 `run_id` 关联应用日志、数据库版本和 LangSmith Trace。

## 9. 首期验收标准

- 用户能完成一趟 2 至 7 天国内旅行从规划到执行的完整流程。
- 已锁定、固定预订和已完成活动在局部重规划中 100% 保留。
- 所有 AI 行程修改必须先展示差异并经用户确认。
- Provider 或模型失败不会损坏已确认行程。
- 路线、天气和 POI 均显示来源及更新时间。
- 估算或过期数据具有明确标识。
- 中断、刷新或切换对话后可以恢复当前 Trip 和待确认补丁。
- 核心流程可在桌面端和移动端完成。
- 路线计算、预算汇总、约束校验和版本提交不依赖 LLM 判断。

## 10. 建议实施顺序

1. 建立 Trip 领域模型、版本和补丁机制。
2. 建立 Provider 接口、真实地图/天气接入、缓存和降级。
3. 重构 PlanningGraph，产出可持久化的首个 Trip 版本。
4. 实现旅行工作区、时间轴、地图联动和结构化编辑。
5. 实现 TripOperationGraph、差异预览和局部重规划。
6. 加入旅中执行、预算账本、事件建议和质量评估集。

该顺序优先建立业务事实和确定性边界，再引入外部数据与 Agent 调整，避免将新的产品能力继续堆叠在不可稳定编辑的聊天结果上。
